# Карточка инцидента incident-card

## Выбранный алгоритм

1. **Чтение JSONL** – файл читается построчно через `bufio.Scanner`, каждая строка парсится в структуру `Event`. Номер строки сохраняется для формирования ссылок на исходные события. Строки дриннее 10 МБ игнорируются. 
2. **Построение индексов** – для быстрого доступа к событиям создаются карты (map):
    - `EventIdIndex` - поиск по `event_id`;
    - `UserIdGroup` - группировка связанных событий по `user_id`;
    - `FileIdGroup` - группировка связанных событий по `file_id`;
    - `DestinationIdGroup` - группировка связанных событий по `destination_id`.
3. **Поиск главного события** – по переданному `event_id` из индекса. Если событие не найдено - программа завершается ошибкой.
4. **Сбор временного контекста главного события** – из всего списка событий выбираются те, которые попадают в временной интервал `[mainTime - before, mainTime + after]` (`before` и `after` – временные окна из запроса). Поддерживаются форматы duration: `5m`, `30m`, `1h` и т.д.
5. **Сбор связанных событий** – из индексов извлекаются события того же пользователя, с тем же файлом и с тем же адресатом (если соответствующие флаги включены в запросе).
6. **Построение таймлайна** – все уникальные события объединяются в один список, удаляются дубликаты (по `event_id`), каждому событию назначается ***роль*** (причина, по которой данное событие попало в карточку) с приоритетом:
   `main_event` > `context_before` / `context_after` > `same_file` > `same_destination` > `same_user`.  
   Затем список сортируется по времени (при равенстве - по `event_id`).
7. **Проверка подозрительных факторов** – каждое правило из YAML-файла проверяется на главном событии. Сработавшие факторы сохраняются и выводятся в соответствующем списке в отчёте. Поддерживаются условия: `equals`, `in`, `contains`, `exists`, `gt`, `gte`, `lt`, `lte`.
8. **Генерация Markdown** – формируется отчёт со всеми разделами (резюме, главное событие, контекст, связанные события, таймлайн, факторы, ссылки).
9. **Генерация JSON** – структурированный вывод в формате JSON (соответствует структуре `Answer`).
10. **Генерация DOT-графа** - если указан флаг `--dot`, строится ориентированный граф связей событий, пользователей, файлов и адресатов. Главное событие выделяется голубым цветом.

## Компромиссы

- **Подозрительные факторы** проверяются только на главном событии, что упрощает реализацию и ускоряют работу, но при дальнейшем развитии проекта возможна реализация проверки на всех событиях таймлайна.
- **Опертатор `contains`** для числового поля `size_bytes` не поддерживается - возвращается `false` с выводом предупреждения. Это связано с тем, что `contains` имеет смысл только для строк и массивов.
- **Индексы в памяти** - весь набор событий загружается в память. Для 1 000 000 событий это ~246 МБ, что удовлетворяет лимиту задания 768 МБ.

## Известные ограничения
- Подозрительные факторы проверяются только на главном событии, а не на всех событиях временной шкалы (соответствует минимальному уровню реализации, но есть возможность расширения).
- При проверке условий подозрительных факторов оператор `contains` для числового поля `size_bytes` не поддерживается и возвращает `false` с выводом предупреждения.
- Утилита не поддерживает многопоточность и предусматривает одновременную работу только с одним файлом.
- Строки длиннее 10 МБ игнорируются с выводом предупреждения.
- При дублировании `event_id` в индекс попадает только первое встречное событе. 

## Результат бенчмарка
Тестирование производительности проводилось на машине:
- **OS:** Windows 11;
- **CPU:** 13th Gen Intel(R) Core(TM) i7-13620H;
- **RAM:** 16GB;
- **Go:** 1.26.4.

Бенчмарки выполнялись на **1 000 000 событий** со структурированными данными (события с повторяющимися связями между пользователями, файлами и адресатами):
| Операция | Время | Кол-во итераций | Память | Аллокации
|---|----|:---:|---|---|
|`BuildIndex`| 541 мс | 3 | 246 МБ | 9063 |
|`GetEventsInTimeRange`| 64 мс | 22 | 0 | 0 |
|`BuildAnswer`| 121 мс | 14 | 0.9 MБ | 104 |
|`BuildTimeline`| 4807 мс | 1 | 405 MБ | 16382 |
|`CheckRules`| 620 нс | 1807767 | 192 Б | 2 |
|`GenerateMarkdownCard`| 147395 нс | 7675 | 67448 Б | 1331 |
|`GenerateDOTGraph`| 223012 нс | 8946 | 117945 Б | 1408 |

Потребление памяти процесса не превышает **405 MБ** для индексации 1 000 000 событий, что соответствует требованиям, где лимит в 768 МБ. Бенчмарк `CheckRules` демонстрирует высокую производительность проверки правил – менее 620 нс на событие.

### Полный сценарий на 1000000 событий

Для проверки соответствия требованию по памяти был выполнен полный цикл: генерация 1000000 событий, построение индекса и сборка карточки. Замеры проводились вручную на машине с характеристиками:
- **OS:** Windows 11;
- **CPU:** 13th Gen Intel(R) Core(TM) i7-13620H;
- **RAM:** 16GB;
- **Go:** 1.26.4.

**Результаты**

|Этап|Время|Писковая память|
|----|-----|---------------|
|Генерация 1000000 событий| ~12 c | ~666 МБ |
|Индексация и отчёт| ~12 c| ~1197 МБ|

Общая пиковая память составила ~1197 МБ, что превышает 768 МБ. Превышение связано с тем, что индексы строятся для всех событий, даже если они не используются в конкретном запросе.

## Возможные улучшения при промышленной доработке

1. **Проверка факторов на всех событиях** - расширить возможнотси механизма для выявления подозрительных факторов не только на главном событии, но и на всех событиях таймлайна для полноты анализа.
2. **Потоковая обработка данных** - для файлов, содержащих более 1 млн событий можно не загружать все данные в память, а обрабатывать их потоково.
3. **Уровни логирования** - добавить поддержку уровней логирования (debug, info, error) для упрощения отладки.
4. **Многопоточность** - реализация параллельного чтения, индексации и построения таймлайнов для ускорения обработки файлов, содержащих более 1 млн событий.
5. **Поддержка других форматов** - на начальном этапе реализована поддержка только JSONL, но будет правильно со временем расширить возможность чтения с поддержанием и других форматов (CSV, Avro и т.д.)
