# Карточка инцидента incident-card

Проект представляет собой CLI-утилиту, которая строит карточку инцидента вокруг одного главного события. 

**Как это работает?**  
Утилита находит событие по `event_id`, собирает контекст (события пользователя до и после, события с тем же файлом, события с тем же адресатом), строит таймлайн, выявляет подозрительные факторы и формирует отчёт в виде карточки Markdown и JSON.

**В чём задача проекта?**  
Проект предназначен для анализа инцидентов в системах информационной безопасности. Вместо ручного просмотра логов происходит автоматическое формирование карточки с ключевой информацией.

## Сборка и запуск

### Требования

- **Go** - версия 1.26.4 и выше ([скачать](https://go.dev/dl/)).
- **Make** (рекомендуется) - для Windows рекомендуется использовать *Git Bash*, который уже включает `make`.
- **Git** - необходим для клонирования репозитория (можно просто скачать ZIP-архив).

### Быстрая сборка (рекомендуемый способ)

```bash
# 1. Клонируете репозиторий (или скачайте ZIP-архив и перейдите к пункту 2)
git clone https://github.com/TaisiaStepanenko/incident-card.git
cd incident-card

# 2. Соберите бинарный файл
make build

# 3. Запустите демо версию на тестовых данных
make demo
```
После выполнения `make demo` в директории `testdata/control/` будут созданы файлы:
- `card.md` - Markdown-карточка инцидента;
- `card.json` - карточка инцидента в JSON-файле.

---

### Ручная сборка без `make`

Если у вас нет `make`, сборку можно осуществить, используя стандартную команду `go`.
#### На Linux / macOS / Git Bash
```bash
go build -o incident-card ./cmd/incident-card/main.go
```

#### На Windows
```bash
go build -o incident-card.exe ./cmd/incident-card/main.go
```

### Запуск после ручной сборки
После сборки запустите утилиту с подкомандой `build`. Пример с минимальным набором флагов:

#### На Linux / macOS / Git Bash
```bash 
./incident-card build --events testdata/control/events.jsonl --event-id evt_12345 --out card.md
```

#### На Windows
```bash 
.\incident-card.exe build --events testdata/control/events.jsonl --event-id evt_12345 --out card.md
```

Для сохранения отчёт и в JSON, добавьте флаг `--json`:
```bash 
.\incident-card.exe build --events testdata/control/events.jsonl --event-id evt_12345 --out card.md --json card.json
```

## Форматы входных данных

Утилита принимает три типа входных файлов: события (обязательный JSONL-файл), запрос (опциональный JSON-файл) и правила подозрительных факторов (опциональный YAML-файл).

---

### Формат события
**Пример строки события (JSON-объект):**
```json
{"event_id":"evt_12345","timestamp":"2026-06-16T10:15:00Z","user_id":"user_017","machine_id":"pc_003","department":"sales","action":"email_send",
"channel":"email","file_id":"file_778","file_name":"client_base.xlsx","file_ext":"xlsx","content_classes":["client_data","personal_data"],
"destination_id":"dst_009","destination_type":"external","destination":"external_email_001","size_bytes":204800,"severity":"high"}
```

#### Обязательные поля
- `event_id` – уникальный идентификатор события;
- `timestamp` – время события в формате RFC3339;
- `user_id` – идентификатор пользователя;
- `machine_id` – идентификатор компьютера;
- `action` – действие (например, email_send, open_file);
- `channel` – канал (local, email, usb, cloud, messenger, printer).

#### Опциональные поля
- `department` – отдел ( sales, hr, finance, dev, legal, support);
- `file_id` – идентификатор файла;
- `file_name` – имя файла;
- `content_classes` – классы содержимого (client_data, personal_data, finance, source_code, legal, none);
- `destination_id` – идентификатор адресата;
- `destination_type` – тип адресата (none, internal, external, usb, cloud, printer);
- `destination` – отображаемое имя адресата;
- `size_bytes` – размер;
- `severity` – важность события (low, medium, high, critical).

---

### Запрос (JSON, опционально)
Файл с параметрами построения карточки. Передаётся через флаг `--request`. Если файл не указан, используются значения по умолчанию.

**Пример `request.json`**
```json
{
  "incident_id": "inc_001",
  "main_event_id": "evt_12345",
  "window_before": "30m",
  "window_after": "10m",
  "include_same_user": true,
  "include_same_file": true,
  "include_same_destination": true,
  "max_events_per_section": 50
}
```

**Поля запроса**

| Поле | Тип | Описание | Возможные значения |
|------|-----|----------|------------------------|
| `incident_id` | `string` | ID карточки инцидента | строка |
| `main_event_id` | `string` | ID главного события | ID из файла событий |
| `window_before` | `string` |  Контекст до события | 5m, 10m, 30m, 1h, 24h |
| `window_after` | `string` | Контекст после события | 5m, 10m, 30m, 1h |
| `include_same_user` | `bool` | Включать события того же пользователя | true, false; по умолчанию `true` |
| `include_same_file` | `bool` | Включать события с тем же файлом | true, false; по умолчанию `false` |
| `include_same_destination` | `bool` | Включать события с тем же адресатом | true, false; по умолчанию `false` |
| `max_events_per_section` | `int` |  Ограничение размера разделов | 1 ... 1000; по умолчанию 50 |

> **Приоритет:** Значения, переданные через CLI-флаги имеют приоритет над значениями из `request.json`

---

### Правила подозрительных факторов (YAML, опционально)
Файл с правилами для выявления подозрительных признаков в событии. Передаётся через флаг `--factors`.

**Пример `factors.yaml`:**
```yaml
factors:
  - factor_id: external_destination
    title:  
    condition:
      field: destination_type
      equals: external
  - factor_id: client_data
    title:   
    condition:
      field: content_classes
      contains: client_data
```

**Поля фактора**

| Поле | Тип | Название |
|:----------|:--------------|--------------|
|`factor_id`|string|Идентификатор фактора|
|`title`|string|Название фактора в отчёте|
|`condition`|object|Условие по полям события|

**Поддерживаемые условия:**
- `equals` – поле точно равно значению. Пример: `equals: external`; 
- `in` – поле входит в список значений Пример: `in: ["high", "critical"]`; 
- `contains` – поле содержит подстроку (для строк) или элемент
(для массивов) Пример: `contains: client_data`; 
- `exists` – поле присутствует и ему присвоено значение Пример: `exists: true`; 
- `gt` – число поля больше значения Пример: `gt: 1048576`; 
- `gte` – число поля больше или равно значению Пример: `gte: 1048576`; 
- `lt` – число поля меньше значения Пример: `lt: 1048576`; 
- `lte` – число поля меньше или равно значению Пример: `lte: 1048576`.
> **Важно:** Оператор `contains` не поддерживается для числового поля `size_bytes`. При попытке его использования программа выведет предупреждение и вернёт `false`. 

---

## Формат выходных данных 
Утилита формирует два вида отчёта: **Markdown** и **JSON**. Оба содержат одну и ту же информацию, но в разных форматах.

---

### Markdown-отчёт

Markdown-карточка (Markdown-файл `card.md`) должна содержать разделы:

1. **Краткое резюме** – краткое описание инцидента;
2. **Главное событие** – информация о главном событии, вокруг которого строится карточка;
3. **Контекст до события** – события, произошедшие в заданном временном окне до главного события;
4. **Контекст после события** – события, произошедшие в заданном временном окне после главного.;
5. **Связанные события пользователя** – все события пользователя;
6. **Связанные файлы** – все события с тем же файлом;
7. **Связанные адресаты** – все события с тем же адресатом;
8. **Временная шкала** – таблица всех уникальных событий, попавших в карточку, в их временной хронологии;
9. **Подозрительные факторы** – список сработавших правил из `factors.yaml`;
10. **Ссылки на исходные события** – для каждого события из таймлайна указывается файл и номер строки, где оно хранится.

**Пример фрагмента Markdown-отчёта:**
```markdown
# Карточка инцидента

__ID инцидента:__ 

## Краткое резюме ##

Пользователь ***user_017*** совершил действие ***email_send*** с файлом ***client_base.xlsx*** в адрес ***external_email_001***.

## Главное событие ##

- __Event ID:__ evt_12345
- __Action:__ email_send
## Контекст до события ##

Подходящих для данного раздела событий не найдено.

## Контекст после события ##

Подходящих для данного раздела событий не найдено.

## События того же пользователя ##

- evt_12345
- evt_12346
- evt_12348
- evt_12351
- evt_12353
- evt_12359
- evt_12365

...
```

---

### JSON-отчёт
Файл `card.json` содержит те же данные, что и Markdown, но в структурированном JSON-формате. Его структура соответствует Go-структуре Answer и включает все перечисленные разделы.

**Пример фрагмента структуры:**
```json
{
  "incident_id":"inc_001",
  "main_event":{"event_id":"evt_12345","action":"email_send"},
  "summary":" user_017  client_base.xlsx   ",
  "context_before":["evt_12000","evt_12100"],
  "context_after":["evt_12400"],
  "same_user_events":["evt_12000","evt_12345","evt_12400"],
  "same_file_events":["evt_12000","evt_12345"],
  "same_destination_events":["evt_12345","evt_12500"],
  "timeline":[
    {
      "timestamp":"2026-06-16T10:10:00Z",
      "event_id":"evt_12000",
      "role":"context_before",
      "user_id":"user_017",
      "action":"create_archive",
      "file_name":"client_base.zip",
      "destination":"",
      "severity":"medium"
    },
    {
      "timestamp":"2026-06-16T10:15:00Z",
      "event_id":"evt_12345",
      "role":"main_event",
      "user_id":"user_017",
      "action":"email_send",
      "file_name":"client_base.xlsx",
      "destination":"external_email_001",
      "severity":"high"
    }
  ],
  "suspicious_factors":["destination_type=external","content_classes contains client_data","archive or file was 
deleted after send"]
}
```

---

### DOT-граф связей
При указании флага `--dot <файл>` утилита генерирует файл в формате DOT, описывающий ориентированный граф связей. Граф включает все события из временной шкалы, а также узлы для пользователей, файлов и адресатов, с которыми эти события связаны. Главное событие выделяется цветом.

---

### Где расположены сгенерированные файлы

- При ручном запуске с флагами `--out` и `--json` файлы создаются в текущей рабочей директории (или по указанному пути);
- При выполнении `make demo` файлы сохраняются в `testdata/control/` с именами `card.md`и `card.json`.

## Примеры команд

### 1. Базовый запуск (только обязательные флаги)

Базовый сценарий содержит флаги, указывающие на: файл событий, ID главного события и выходной Markdown-файл.

```bash 
./incident-card build --events events.jsonl --event-id evt_12345 --out card.md
```

Для сохранения отчёта в JSON необходимо добавить флаг --json:
```bash 
./incident-card build --events events.jsonl --event-id evt_12345 --json card.json
```

Так же возможно сохранение в обоих форматах. Для этого необходимо задать оба параметра.
```bash 
./incident-card build --events events.jsonl --event-id evt_12345 --out card.md --json card.json
```

### 2. Запуск с JSON-запросом и правилами факторов
Запуск с файлами запроса и YAML-файлом с правилами подозрительных факторов.
```bash 
./incident-card build \
    --events events.jsonl \
    --request request.json \
    --factors factors.yaml \
    --out card.md \
    --json card.json
```

В данном случае основные параметры (`main_event_id`, `window_before`, `window_after` и др.) берутся из `request.json`, а правила фактором из `factors.yaml`.

### 3. Запуск с переопределением параметров через CLI (приоритет над request.json)

Если вы хотите переопределить часть параметров из `request.json`, передайте их через CLI. CLI-флаги имеют более высокий приоритет.

```bash
./incident-card build \
    --events events.jsonl \
    --request request.json \
    --event-id evt_67890 \
    --before 1h \
    --out card.md
```
В данном примере значение `--event-id` и `--before` будут использованы вместо тех, что указаны в `request.json`. Остальные параметры (например, window_after, флаги включения) будут определяться значением, переданным в `request.json`.

### 4. Запуск без сохранения
Если при запуске не был указан параметр `--out`, программа не сохранит Markdown, но при этом всё равно выполнит все вычисления. Аналогично без `--json` не будет сохранён JSON.

```bash
./incident-card build --events events.jsonl --event-id evt_12345
```
При запуске команды выше программа выведет сообщение, что для сохранения отчёта необходимо указать `--out` и/или `--json`.

### 5. Построение DOT-графа связей
Для визуализации связей между событиями, пользователями, файлами и адресатами используйте флаг `--dot`:

```bash
./incident-card build --events events.jsonl --event-id evt_12345 --dot graph.dot
```

Это создаст файл `graph.dot` в формате DOT, который можно преобразовать в изображение с помощью утилиты Graphviz:

```bash
dot -Tpng graph.dot -o graph.png
```

На полученном графе главное событие будет выделено голубым цветом, пользователи - эллипсами, файлы - заметками, адресаты - цилиндрами. Рёбра подписаны в зависимости от узлов, которые они связывают: `performed by` для связи событие - пользователь, `uses file` для связи событие - файл, `sends to` для связи событие - адресат.

### Генерация тестовых событий
Для создания набора событий по заданному сценарию используйте команду `generate`

```bash
./incident-card generate --count 1000 --scenario external_send --out generated_events.jsonl --seed 42
```

#### Параметры

| Флаг | Описание | Значение по умолчанию |
|:----------|:--------------|--------------|
|`--count`|Количество генерируемых событий|100000|
|`--scenario`|Сценарий: `external_send`, `usb_copy`, `cloud_upload`|`external_send`|
|`--out`|Выходной JSONL-файл, куда будут записаны сгенерированные события|обязательное поле|
|`--seed`|seed генератора для воспроизводимости|42|

### Сценарии:
- `external_send` - пользователь открывает/архивирует файл с клиентскими данными и отправляет его во внешний канал;
- `usb_copy` - пользователь работает с файлом и копирует его на USB;
- `cloud_upload` - пользователь открывает или собирает файл и загружает его в облако.

Сгенерированные события образуют мини-сюжет: главное событие, события до/после, тот же пользователь и файл, что позволяет тестировать все функции карточки инцидента.

## Краткое описание алгоритма

Работа утилиты заключается в следующий последовательных шагах:

1. **Чтение JSONL** – файл читается построчно, каждая строка парсится в структуру `Event`. Номер строки сохраняется для формирования ссылок на исходные события.
2. **Построение индексов** – для быстрого доступа к событиям создаются индексы:
    - `by_event_id` – для быстрого поиска главного события;
    - `by_user_id`, `by_file_id`, `by_destination_id` – для группировки связанных событий.
3. **Поиск главного события** – по переданному `event_id` находится событие, вокруг которого будет строиться карточка.
4. **Сбор временного контекста главного события** – из всего списка событий выбираются те, которые попадают в временной интервал `[mainTime - before, mainTime + after]` (`before` и `after` – временные окна из запроса).
5. **Сбор связанных событий** – из индексов достаются события того же пользователя, с тем же файлом и с тем же адресатом (если соответствующие флаги включены в запросе).
6. **Построение таймлайна** – все собранные события объединяются в один список, удаляются дубликаты (по `event_id`), каждому событию назначается ***роль*** (причина, по которой данное событие попало в карточку).  
    Приоритет ролей:  
   `main_event` > `context_before` / `context_after` > `same_file` > `same_destination` > `same_user`.  
   Затем список сортируется по времени (при равенстве — по `event_id`).
7. **Проверка подозрительных факторов** – каждое правило из YAML-файла проверяется на главном событии. Сработавшие факторы сохраняются и выводятся в соответствующем списке в отчёте.
8. **Генерация Markdown** – формируется отчёт со всеми разделами (резюме, главное событие, контекст, связанные события, таймлайн, факторы, ссылки).
9. **Генерация JSON** – структурированный вывод в формате JSON (соответствует структуре `Answer`)

## Известные ограничения
- Подозрительные факторы проверяются только на главном событии, а не на всех событиях временной шкалы (соответствует минимальному уровню реализации, но есть возможность расширения).
- При проверке условий подозрительных факторов оператор `contains` для числового поля `size_bytes` не поддерживается и возвращает `false` с выводом предупреждения.
- Утилита не поддерживает многопоточность и предусматривает одновременную работу только с одним файлом.

## Результаты контрольного запуска
После выполнения `make demo` на контрольном наборе данных (21 событие из файла `testdata/control/events.jsonl`) формируется карточка инцидента для события `evt_12345` с временными окнами `30m` до события и `10m` после. Результаты сохраняются в `testdata/control/card.md` и `testdata/control/card.json`. Данные файлы следует использовать для проверки корректности форматов и содержания.
